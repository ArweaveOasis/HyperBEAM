# Review Cycle 4

The fourth review cycle was the “long tail” pass: the direct callers, helper modules, and tests that were touched because the store contract moved, but which are not themselves store backends. This is usually where the most subtle quality problems hide. The big architectural work is already done by that point, so what remains are the edits that are easy to justify in the moment and easy to regret later because they widen behavior, weaken tests, or change runtime semantics without carrying their weight. This cycle found two concrete examples of that pattern and improved both of them.

I started by rereading the smaller diff files with a bias toward skepticism. In modules like `dev_arweave`, `dev_location`, `dev_secret`, `hb_private`, `dev_match`, `dev_process_cache`, and `dev_scheduler_cache`, most of the edits are straightforward fallout from the new store API. Calls that used to look like `hb_store:write(Store, Path, Value)` now need to pass a request map and `Opts`. Calls that used to compare against bare `not_found` now need to handle `{error, not_found}`. Path construction moved from `hb_store:path/2` to `hb_path:to_binary/1`. For many of these modules, the best outcome of review was simply to confirm that the changes are necessary and already fairly minimal. I did not want to create churn just to make the review loop feel busy.

One file that deserved a much harder look was `dev_node_process`. The first draft contained two kinds of changes there. One was clearly necessary: the tests were updated to use unique process names rather than a fixed singleton test name, which prevents stale local-name and cache state from bleeding across runs. That is a clean fix to a real isolation problem. The other change was more suspicious: the runtime spawn path had been modified to force `Signed` through `hb_cache:ensure_all_loaded/2`, producing `LoadedSigned`, and then using that loaded form for process ID calculation, scheduler initialization, events, and the returned process value.

That second change was a perfect candidate for review-driven rollback because I already knew, from the earlier debugging sequence, that it had not actually fixed the failing tests. The failures cleared only after the test names became unique per run. In other words, the `LoadedSigned` runtime rewrite was not the root-cause fix; it was an unsuccessful attempt that happened to remain in the patch. Leaving that in would have widened runtime behavior in a core process-spawn path for no proven benefit. I reverted it. `spawn_register/2` now goes back to using `Signed` directly, while the unique-name test helper changes remain. After making that rollback, I reran the `dev_node_process` tests along with several adjacent process/cache modules, and they passed cleanly. This is exactly the kind of refinement I want from these cycles: smaller diff, same coverage, less unexplained runtime change.

The other major area I focused on in this cycle was `dev_lua`, because one of the diffs stood out as too blunt to leave unexamined. The direct Lua benchmark assertion had been weakened from `Iterations > 10` to `Iterations >= 1`. Even before running anything, that change looked suspicious. It is the kind of edit that can keep CI green while quietly removing useful signal. Since the user explicitly noted that the Lua process benchmarks might actually be faster on this branch and asked me to preserve that improvement through the PR, I treated this as a review priority rather than a harmless test tweak.

I first tried the most surgical rescue: warm up the direct benchmark path once, then restore the stronger threshold. That did not work. Even after the warm-up call, the direct benchmark still only completed one execution in the 250ms measurement window in this environment. That told me something important: for this machine and current runtime, `direct_benchmark_test` is dominated by startup or fixed overhead and is not acting like a reliable throughput regression test. Pretending otherwise by leaving a high threshold in place would just create instability. On the other hand, leaving the relaxed `>= 1` assertion without any compensating signal would be too weak.

So I changed strategy. I kept the direct benchmark as a smoke-style benchmark with the `>= 1` assertion, which matches the behavior we can actually reproduce here, and I moved the meaningful guardrails to the process benchmarks that the user specifically called out. In `pure_lua_process_benchmark/1`, I now capture the measured execution time in milliseconds and assert that the final “compute now” step completes within 500ms. In `aos_process_benchmark_test_/0`, I do the same and assert completion within 250ms. These thresholds are intentionally conservative relative to the observed results in this environment, which were around 88–173ms for the pure Lua path and 41–76ms for the HyperAOS path across the validation runs. The goal is not to freeze the exact benchmark number. It is to preserve a real performance floor in the process paths that matter, without introducing machine-fragile micro-benchmarks.

This shift makes the benchmark coverage more honest. The direct benchmark still provides visibility into that path and will fail if it stops working entirely, but it no longer pretends to enforce a throughput property that this environment does not sustain. Meanwhile, the process benchmarks now carry explicit expectations that match the improvement the user noticed. I reran the `dev_lua` module after these edits and confirmed that the new timing guards pass with healthy margin.

Beyond those two concrete edits, I also reviewed the remaining long-tail modules for hidden regressions and decided not to “improve” them unnecessarily. `dev_process_cache`’s move from `resolve + type` probing to a direct `hb_store:read/3` presence check is a real simplification and aligns with the new composite-read model. `dev_scheduler_cache`’s path construction changes are straightforward `hb_path` migrations, and its continued use of `not_found` at its own API boundary is acceptable because that module is not exposing the raw store adapter surface. `dev_secret`, `hb_private`, and `dev_location_cache` all now pass canonical request maps into the store layer where appropriate. Those edits looked proportionate to the refactor and did not reveal any extra runtime behavior changes worth reverting.

Validation for this cycle was broader than the earlier focused slices because the changes touched both tests and runtime behavior in the process-and-Lua areas. I ran `dev_node_process`, `dev_process_cache`, `dev_scheduler_cache`, `dev_location`, `dev_secret`, `hb_private`, and `dev_lua` together. That suite passed cleanly, including the new process benchmark assertions and the reverted `dev_node_process` runtime path.

The overall effect of cycle four is that the branch now feels less accidental at the edges. The test isolation fix in `dev_node_process` stays, but the unrelated runtime mutation is gone. The Lua benchmarks now say something defensible: one path is informational, and the process paths carry actual regression guards. That is a much better place to be before the final diff-wide pass, because it means the remaining review is less about cleaning up after experimentation and more about confirming that the patch as a whole is as intentional as it looks.
