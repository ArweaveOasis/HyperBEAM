# Review Cycle 3

The third review cycle was dedicated to RocksDB, but in a deliberately constrained way. The user guidance here was not “make RocksDB great again.” It was much narrower: treat RocksDB as out of scope except for basic passing `hb_store` coverage, and revert any spillover complexity introduced while trying to make it behave like a first-class backend again. That changed how I approached the review. Instead of asking what features or polish I could add, I asked a much stricter question: what compatibility scaffolding is still sitting in `hb_store_rocksdb` after the refactor, and do we actually need it to support the limited surface we intend to keep?

I began by reading the full module from the top rather than diff-skimming. That mattered because the RocksDB file is one of the easiest places to get fooled by stale structure. It still contains legacy-style helper names, a test-heavy lower half, and a mixture of new request-map entry points and older helper patterns. The first thing I confirmed is that the high-level shape is right for the refactor: the exported surface is now `start/1`, `start/3`, `stop/1`, `stop/3`, `reset/1`, `reset/3`, `read/3`, `write/3`, `list/3`, `link/3`, `group/3`, `type/3`, and `resolve/3`. That aligns with the new store contract, which means the remaining cleanup should be about deleting leftover compatibility layers, not redesigning the module.

The compiler warnings from the RocksDB profile were especially helpful in this cycle because they pointed straight at code that had become structurally redundant. `read_path/2`, `read/2`, `resolve/2`, and `type/2` were still present even though the tests and all public callers had already moved to the canonical request-map forms. The module was effectively carrying two APIs: the one we want, and a private echo of the old one that no longer served any runtime purpose. That is exactly the kind of accidental maintenance burden that should be removed when a backend is explicitly being narrowed in scope.

I did not remove those helpers blindly. For each one, I checked whether it was part of the public exported surface, whether it was still referenced internally, and whether any embedded tests depended on it. The answer was revealing. `read_path/2` was only being used by `read/2`, and `read/2` itself was no longer used. `resolve/2` was just a thin wrapper over `resolve_path/2`, but nothing needed the wrapper anymore. `type/2` existed only to bounce into `type_path/2`, which was already the real implementation. In other words, these functions were not keeping behavior alive; they were keeping old naming patterns alive. That made the decision easy. I deleted the unused wrapper layers and let the already-existing internal helpers (`resolve_path/2`, `type_path/2`, and the direct `do_read/2`-based logic inside `read/3`) carry the implementation.

The `list` path deserved a bit more care. Unlike the others, `list/2` was still indirectly participating because `list_path/2` handled link targets by calling `list(Opts, LinkedPath)`. That is a classic sign of a wrapper surviving mostly because the code fell into referencing it from an older internal call site. I changed that internal recursion to call `list_path/2` directly instead. Once that was done, the old `list/2` wrapper could go away cleanly. That is a small change, but it improves the module in exactly the way this cycle was aiming for: fewer surfaces, less indirection, no loss of behavior.

While I was in the RocksDB module, I also looked for “same result, simpler code” opportunities that would not broaden scope. One good example was `group/3`. The first draft had two clauses for it, one matching a store map that included `<<"name">>` and one matching everything else, but both clauses did the exact same work: call the gen_server with a normalized binary path. Carrying two clauses there does not improve correctness or clarity. It just makes the module look more special-case-driven than it really is. I collapsed those into a single clause. That kind of simplification is especially worthwhile in a backend we are intentionally not investing in deeply right now.

There was one small correction during the cycle that was worth catching in the memo because it reflects the value of the loop itself. After removing the legacy `list/2` wrapper, the module still had a stale `-spec list(Opts, Path)` declaration describing a function that no longer existed. The narrow RocksDB test run failed immediately with “spec for undefined function list/2.” That was a good failure: it meant the cleanup was real enough to break stale declarations instead of merely shuffling code around. I updated the spec to describe `list_path/2`, reran the suite, and the RocksDB slice passed again.

Importantly, I did not use this cycle to chase the remaining “behaviour hb_store undefined” warnings that appear in the RocksDB profile compile. Those warnings also appear for other store modules under the `as rocksdb` build and are about compile order or profile-specific behavior visibility, not about unused compatibility logic inside RocksDB itself. Trying to solve that here would have expanded the scope into build-configuration mechanics, which is precisely the kind of distraction the user asked me to avoid. The point of this cycle was to keep RocksDB minimal and sufficient, not to turn it into a fresh subsystem project.

I also resisted the urge to rewrite the embedded test names that still mention `type/2`, `resolve/2`, and `list/2` in their descriptions even though the tests now call the new request-map forms. I noticed the mismatch, but changing those names is cosmetic and does not improve the backend’s supported surface. It would also add more diff noise to a file that already changed significantly in the main refactor. Given the user’s emphasis on surgicality, leaving those labels alone was the better call for now.

For validation, I reran the narrow RocksDB coverage the user explicitly said was sufficient: `rebar3 as rocksdb eunit --module=hb_store,hb_store_rocksdb` with the CMake path override and a dedicated port. The first rerun failed for the right reason, due to the stale spec, and the second rerun passed cleanly after fixing it. That is exactly the shape I want from a review cycle: one or two targeted improvements, immediate validation, no uncontrolled scope growth.

What this cycle improved is subtle but meaningful. The RocksDB backend now more honestly reflects its current role in the branch. It supports the canonical store API, passes the basic `hb_store` and backend tests we care about, and carries less stale compatibility scaffolding from the pre-refactor world. That makes future maintenance less confusing. An engineer reviewing the file tomorrow morning will see a backend that participates in the new contract, not a backend that appears to support two overlapping contracts because the migration stopped halfway. Given the explicit instruction to keep RocksDB limited, that is the right finishing point for this part of the patch.
