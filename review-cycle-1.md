# Review Cycle 1

This first review cycle focused on the smallest but most important layer of polish: the core store wrappers and the behavioral contract that every other module now depends on. After the first draft commit, I deliberately did not start by chasing broad architectural questions. Instead, I re-read the changed code with a stricter lens for mechanical consistency, because that is where a refactor like this can still feel rough even when the tests are already green.

The first thing I checked was whether the branch actually honored the instruction to eliminate `{ok, ok}`. I had already removed the literal tuple in many direct call sites, but a careful grep revealed a subtler version of the same problem: several admin-style wrapper heads were still constructing the tuple dynamically. In other words, code such as `start(Store, _Req, _Opts) -> {ok, start(Store)}.` did not contain the literal text `{ok, ok}`, so the earlier search did not catch it, but it still produced the same absurd result whenever `start/1` returned `ok`. That was exactly the kind of gap this review loop is supposed to flush out. I corrected that in the filesystem store, the volatile store, the LMDB store, and the multi-store wrapper by returning the underlying admin result directly instead of re-wrapping it.

In `hb_store_fs`, I reviewed the start, stop, and reset wrappers first because the filesystem store is one of the simplest backends and therefore a good place to verify intent. Here the right answer was not to add a normalization helper or another layer of abstraction. The right answer was simply to let `start/3`, `stop/3`, and `reset/3` call the existing unary helpers and return their result as-is. That keeps the module readable, keeps the contract obvious, and avoids manufacturing a tuple that carries no information. I also re-read the rest of the diff in that module while I was there. The broader request-style changes still look justified. The use of `hb_path:to_binary/1` in place of the old `hb_store:join/1` is a real improvement, and the `read_path/1` composite return path is necessary to support the new `hb_cache` traversal model.

In `hb_store_volatile`, the only code change in this cycle was the `stop/3` wrapper, but I spent time reviewing more than that. I checked whether `reset/3` was already returning plain `ok` and confirmed that it was. I also re-evaluated the choice to model group reads as `{composite, Keys}` directly from `read_resolved/3`. That remains correct because it brings the volatile backend into alignment with the new store-wide read semantics without forcing `hb_cache` to special-case this backend. I briefly considered whether the path joining logic in `join_path/2` and `next_group_path/2` should be compacted further, but the current form is better than the old version because it clearly encodes the root-group exception and no longer depends on the removed `hb_store:join/1`.

In `hb_store_lmdb`, this cycle was mostly about resisting the temptation to over-edit. The LMDB module saw a large migration in the main refactor, so it is easy to keep “improving” it just because it has many changed lines. I re-read the batch `write/3` and `link/3` paths, the direct helpers, and the public request-form heads. The main concrete issue I found was the same admin-wrapper tuple problem in `stop/3` and `reset/3`, which I fixed. I deliberately did not rewrite the batching structure or the direct helper layout in this cycle, even though there are places where the code could be made slightly denser, because the current split between direct binary-path operations and request-map entry points is actually doing useful work. It makes the public contract explicit while keeping the low-level operations testable and readable. I also verified that the long `write/3` edoc remains intact, because preserving that documentation was part of the user’s feedback and it still accurately describes the asynchronous LMDB writer behavior.

In `hb_store_multi`, the same no-value return-shape issue was present in `stop/3` and `reset/3`, so those wrappers now return `ok` directly. I spent some time reading the surrounding code because this module is especially important to the new inferred-store contract. The multi-store backend is the place where a small inconsistency could turn into very confusing behavior for the rest of the system. What I wanted to confirm here was that the admin operations still behave as all-store side effects, while the read-like operations still behave as first-viable-result selectors, and the write-like operations still respect confirmation thresholds. That structure remains sound. The wrapper cleanup actually makes the admin behavior a little clearer because success is now represented by the same plain `ok` shape used elsewhere, instead of a synthetic tuple that looked like data-bearing success but was not.

In `hb_store`, I looked for code that survived the large dispatcher rewrite but no longer had a job to do. That is where I found `call_all/3`. It had become genuinely dead code after the new `admin_call/4` and request-oriented dispatch flow settled into place. Removing it was not just about silencing a compiler warning. Dead helpers are a tax on future review because they imply alternate flows that no longer exist. In a core module like `hb_store`, that extra cognitive load matters. I verified with ripgrep that there were no remaining references before deleting it. I did not try to refactor the larger dispatcher logic in this cycle because the current implementation is doing real coordination work: normalizing results, handling retries on `{failure, Reason}`, skipping on `{error, Reason}`, and preserving the intended multi-store fallback behavior.

I also used this cycle to reassess the “RocksDB out of scope” instruction. The focused test run still emits a few RocksDB warnings, but importantly those warnings are about unused compatibility helpers and compile-order behavior checks, not about the correctness of the core refactor in the default runtime. I did not address those warnings in this cycle because they are localized, non-blocking, and better handled in a dedicated RocksDB-focused pass rather than mixed into a general wrapper cleanup. That will let me evaluate whether each RocksDB helper is still needed for the limited supported surface, instead of making rushed deletions under the banner of tidiness.

The validation strategy for this cycle was intentionally narrow. After changing the admin wrappers and removing `call_all/3`, I ran the focused store-and-cache slice covering `hb_store`, `hb_cache`, `dev_cache`, `dev_process_cache`, `hb_store_volatile`, `hb_store_multi`, and `hb_store_lmdb`. That suite passed cleanly. I did not rerun the full 3000-plus test default suite immediately after this cycle’s edits because the changes were tightly scoped and the focused slice gives fast feedback on exactly the surfaces I touched. I will reserve the heavier reruns for later cycles when broader edits accumulate.

The most important outcome of this cycle is not the line count reduction. It is that the branch now says what it means more consistently. No-value admin success is represented as `ok`, not as a tuple pretending to carry a payload. Dead coordination code is gone. The changed modules still preserve the larger refactor’s intent: device-style store entry points, canonical request maps, normalized result categories, and a read path that can distinguish simple values from composites. That makes the patch easier to reason about in the morning review, and that clarity is exactly what this loop is for.
